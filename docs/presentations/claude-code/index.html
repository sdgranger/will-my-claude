Tool Call — LLM이 행동하는 방법

LLM만 있다면...

텍스트 입력 → 🧠 추론 → 텍스트 출력

파일을 읽거나, 코드를 실행하거나, 무언가를 저장할 수 없다

Tool Call이 있다면...

LLM이 "이 도구를 이 인자로 실행해줘"라는 JSON을 출력

Claude Code가 실제로 실행하고 결과를 다시 LLM에게 전달

{
  "type": "tool_use",
  "name": "Read",
  "input": { "file_path": "/src/auth.ts", "limit": 50 }
}

Tool Call 실행 사이클

sequenceDiagram participant U as 👤 사용자 participant CC as ⚡ Claude Code participant LLM as 🧠 LLM participant T as 🛠 도구 U->>CC: "auth.ts 리뷰해줘" CC->>LLM: 메시지 전달 LLM->>CC: tool_use: Read(auth.ts) CC->>T: 파일 읽기 실행 T->>CC: 파일 내용 반환 CC->>LLM: tool_result 전달 LLM->>CC: 리뷰 텍스트 생성 CC->>U: 최종 응답 출력

LLM은 직접 파일을 읽지 않는다 — Claude Code가 실행하고 결과를 전달한다

# Project Context

## Stack
- Next.js 14 (App Router)
- Prisma + PostgreSQL
- Vitest for testing

## Conventions
- 한국어로 커밋 메시지 작성
- 테스트 없이 PR 금지
- .env 파일 절대 커밋 금지

## Commands
- `pnpm test` — 테스트 실행
- `pnpm dev` — 개발 서버

도구(Tools)란?

🧠  +  🛠

LLM + 도구 = 행동할 수 있는 에이전트

도구 없음

텍스트를 읽고
텍스트를 출력할 뿐

도구 있음

파일 읽기, 코드 수정,
명령 실행, 검색

Claude Code에는 수십 가지 도구가 있고, LLM이 필요에 따라 선택해서 호출한다

파일 도구

코드베이스를 읽고 수정하는 핵심 도구들

Read

파일 내용 읽기 (라인 범위 지정 가능)

LLM이 코드를 이해하기 위한 첫 번째 단계

Write

파일 생성 / 전체 덮어쓰기

새 파일을 만들거나 완전히 새로 쓸 때

Edit

특정 텍스트를 찾아 교체 (정밀 수정)

변경된 부분만 diff로 표시 → 리뷰가 쉬움

Glob

파일 패턴으로 경로 검색

**/*.test.ts 처럼 와일드카드 지원

실행 도구 — Bash

Bash

모든 쉘 명령을 실행할 수 있는 가장 강력한 도구

주요 활용

  • 테스트 실행 (npm test)
  • git 조작
  • 빌드 / 린트
  • 패키지 설치

⚠️ 주의

가장 영향력이 큰 도구
실행 전 사용자 승인 필요
권한 범위 설정이 중요

검색 도구 — Grep

Grep

코드베이스에서 패턴을 검색 (ripgrep 기반)

정규식 지원, 파일 타입 필터, 컨텍스트 라인 수 설정

LLM이 Grep을 쓰는 이유

전체 파일을 읽지 않고 필요한 부분만 찾아서
컨텍스트 윈도우를 효율적으로 사용

// LLM의 tool_use 예시
{ "name": "Grep", "input": { "pattern": "useAuth", "type": "ts", "output_mode": "files_with_matches" } }

도구 권한 모델

모든 도구가 자동으로 실행되지는 않는다

✓ 자동 허용

  • Read, Grep, Glob
  • 읽기 전용 도구들
  • 부작용 없는 작업

⚡ 사용자 승인 필요

  • Bash (명령 실행)
  • Write, Edit (파일 수정)
  • 외부 시스템 접근

설정으로 조절 가능: allowedTools, autoApprove 설정으로
신뢰하는 도구는 자동 허용, 민감한 도구는 항상 확인

에이전트(Agent)란?

도구 + 실행 루프 = 에이전트

목표가 달성될 때까지 스스로 반복한다

단순 LLM 호출

질문 1개 → 응답 1개
끝

에이전트

목표 확인 → 계획 → 실행
→ 결과 확인 → 다음 단계 → ...

멀티에이전트(Multi-Agent)란?

에이전트가 다른 에이전트를 도구로 호출하는 구조

Claude Code의 Agent 도구를 통해 서브에이전트를 생성·실행

Orchestrator

전체 목표를 이해하고
하위 작업을 분배

Subagent

하나의 독립적인 작업에
집중해서 실행

각 서브에이전트는 독립적인 컨텍스트 윈도우를 가진다 → 서로 간섭 없이 병렬 실행 가능

# TDD 워크플로우

## 규칙
1. 먼저 실패하는 테스트를 작성
2. 테스트가 실패하는지 확인
3. 통과하는 최소한의 코드 작성
4. 테스트 통과 확인
5. 리팩토링 후 커밋

## 절대 하지 말 것
- 테스트 없이 구현 코드 작성
- 구현 후 테스트 추가

스킬 예시

TDD 스킬

기능 구현 시

테스트 먼저 → 실패 확인 → 구현 → 통과 확인 → 커밋 순서 강제. "구현 먼저 하고 테스트 추가"를 막는다

디버깅 스킬

버그 발생 시

증상 파악 → 가설 수립 → 검증 → 수정 → 재발 방지 순서로. 즉흥적 수정 대신 체계적 접근 강제

브레인스토밍 스킬

새 기능 설계 시

질문 하나씩 → 요구사항 명확화 → 설계 제안 → 승인 후 구현. 코드 먼저 짜는 것을 막는다
